# University Food Map 设计文档

## 1. 版本定位

`University-Food-Map.html` 是在原版基础上重新整理的一版“更克制、更偏真实产品首页”的大学城美食网页原型。

这版的核心目标不是展示更多模块，而是回答一个更实际的问题：

**用户打开页面后，能不能在 10 秒内理解产品，并开始决定今天吃什么。**

## 2. 为什么要新开一版

原版已经具备：

- 较完整的视觉系统
- 大学城认知地图
- 场景化推荐逻辑
- 可直接本地打开与部署

但在竞争性评审中，有几条意见值得吸收：

1. 默认展示的信息有点多
2. 输入框虽然已经提到首屏，但还可以更像真正的一级入口
3. “AI” 文案容易被误解为夸大能力
4. 筛选项默认全展开，会增加用户的首次理解成本

因此本版不替代原版，而是形成第二份交付：

- 保留原版的完成度和地图能力
- 同时做一版更轻、更聚焦、更适合评审快速理解的版本

## 3. 本版设计原则

### 原则一：先输入，再筛选

页面的第一个动作应该是“说一句需求”，而不是先让用户理解全部筛选器。

因此本版把：

- 自然语言输入框
- 快捷 prompt
- 一键随机推荐

放在最前面；

而把预算、人数、距离、口味、情绪等多维筛选收进“展开更多偏好”。

### 原则二：减少默认复杂度

不是删除能力，而是默认先收起，让用户先完成一次决策，再决定要不要继续细化。

### 原则三：保留地图，但更准确地定义地图

本版继续保留地图能力，但明确把它定义为：

**大学城认知地图**

而不是导航地图。

它服务的不是“精确路线规划”，而是：

- 理解学校与商圈的关系
- 理解餐厅分布
- 激发跨校探索和浏览欲望

### 原则四：降低“伪 AI”风险

本版保留智能推荐体验，但在表述上更克制：

- 使用 `智能推荐分析中`
- 使用 `推荐把握`
- 使用 `推荐依据`

而不是过度强调“真正 AI 正在思考”。

## 4. 结构设计

本版主要分为 3 层：

### 4.1 一级入口：首页决策区

目标：

- 一打开就能看懂
- 一输入就能得到结果
- 一眼知道核心任务是“今天吃点啥”

内容包括：

- 大标题
- 自然语言输入框
- 快捷需求词
- 一键随机推荐
- 可折叠高级偏好
- 主推荐 + 2 个备选

### 4.2 二级入口：认知地图

目标：

- 展示学校、商圈、餐厅之间的空间关系
- 在筛选学校时形成明显主次
- 用热力和聚类增强“发现感”

地图组成：

- 学校片区底图
- 商圈标签
- 主路与商圈带
- 热门商圈热力层
- 聚类提示
- 餐厅点位

### 4.3 三层内容：学生榜单

目标：

- 让用户在“不想输入”时也能快速获取灵感
- 用学生任务而非评分高低组织内容

榜单包括：

- `10 分钟能吃上饭`
- `宿舍聚餐不翻车`
- `晚自习后还来得及`

## 5. 本版与原版的主要差异

| 维度 | 原版 | 本版 |
| --- | --- | --- |
| 入口形态 | 首屏已聚焦，但筛选信息相对完整 | 更像“只做一件事”的首页 |
| 筛选方式 | 默认展示较多维度 | 默认折叠高级偏好 |
| AI 表达 | 智能感更强 | 更克制、更稳妥 |
| 地图定位 | 已是认知地图方向 | 文案与定位进一步统一 |
| 评审适配性 | 完整度更强 | 第一眼理解成本更低 |

## 6. 为什么仍然保留单文件

本版依然采用单文件静态交付，原因不是偷懒，而是面向当前任务做出的策略选择：

1. 双击即可打开
2. 不依赖 `npm install`
3. 便于网页上传部署
4. 适合产品题评审快速查看

在真实项目里当然可以拆成工程化结构，但在这道题里，交付成功率比工程拆分优先级更高。

## 7. 本版最想证明的事

这版不是要证明“技术更复杂”，而是要证明：

**在大学城美食地图这个题目里，好的产品设计应该先降低用户决策成本，再提供探索能力。**

所以这版最重要的设计选择是：

- 让输入框成为一级入口
- 让筛选变成辅助
- 让地图继续保留，但不抢首页主任务
